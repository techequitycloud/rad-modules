## 2024-05-18 - Avoid timestamp() triggers in null_resources for performance
**Learning:** In the Bank_GKE and MC_Bank_GKE modules, the `null_resource.download_bank_of_anthos` uses `always_run = timestamp()` as a trigger. This forces a re-download and re-extraction of the Bank of Anthos application archive on *every* `terraform apply`, slowing down deployments significantly, and can also lead to a two-apply state issue where the resource forces replacement of downstream dependencies.
**Action:** Replace dynamic `timestamp()` triggers with static triggers (like version or path) and use idempotent bash scripts in the `local-exec` provisioner (e.g., checking if the file exists before downloading) or just use `always_run = "true"` combined with bash idempotency if it needs to run unconditionally but shouldn't dirty the state tree. Wait, `timestamp()` dirties the state tree. In Terraform, changing triggers replaces the null_resource. If we want it to run on apply without replacing, we can just use `always_run = true` as a trigger, wait no, that's just a static string, it won't force run unless we use `terraform apply -replace`. Actually, `null_resource` does not run again if triggers don't change. So if we just use `version` and `download_path` as triggers, it will only download when those change. Wait, if someone manually deletes the folder, Terraform won't know. To handle that, we could use `manifests_missing = fileexists(...) ? "exists" : timestamp()` but that has the two-apply issue. So just rely on the static triggers. The `version` and `download_path` triggers are sufficient, as Terraform expects the user to not mess with the local disk, or if they do, they can use `-replace`. Let's just remove `always_run = timestamp()`.
