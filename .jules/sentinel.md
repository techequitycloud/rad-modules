## 2025-02-24 - [Over-permissioned GKE Service Accounts]
**Vulnerability:** GKE service accounts (`gke_sa` and `gke_standard`) in `modules/Istio_GKE`, `modules/Bank_GKE`, and `modules/MC_Bank_GKE` are granted the `roles/storage.objectAdmin` role, giving them full administrative control over all objects in all buckets within the project. Additionally, `modules/Bank_GKE` and `modules/MC_Bank_GKE` grant both `roles/storage.objectAdmin` and `roles/storage.objectViewer` simultaneously, which is redundant.
**Learning:** Default node service accounts often only need read access to pull images (from GCR/Artifact Registry) or minimal access. Granting `objectAdmin` to a GKE node pool service account allows any pod on that node pool (which can assume the node's identity) to read, overwrite, or delete *any* object in *any* GCS bucket in the project. This is a severe violation of the least-privilege principle, particularly for a general-purpose cluster running applications like Bookinfo or Bank of Anthos.
**Prevention:** Always scope GKE service accounts to the minimum required roles. For pulling images, use `roles/artifactregistry.reader` (which is already included) and `roles/storage.objectViewer` if GCR is still used. Never use `objectAdmin` unless specifically required and scoped to a single bucket using a bucket-level IAM binding, not a project-level binding.
